### 2026-07-19：Android SQLite 残留事务稳定性修复

- 根据 `img/issue/报错.jpg` 定位到数据更新时原生 SQLite 报错：`BeginTransaction: Already in transaction`。
- 所有原生多语句写入统一经过 `SerialTaskQueue`，普通 `run()` 禁止隐式开启插件事务，避免嵌套事务。
- 同步日志、标签、单笔账单等独立写入也进入同一个队列，禁止在比赛或赛果批量事务执行期间插入原生写操作。
- 数据库关闭操作同样进入写队列，确保应用关闭连接前等待正在执行的原生写入完成。
- 开始事务前检查并清理残留状态；插件状态检测失真时执行无条件安全回滚并重试。
- 连续重试仍失败时先关闭并重开连接；若包装连接仍残留事务，则从 Capacitor 原生连接字典中彻底关闭并重新创建同一数据库连接后再开启事务。
- 禁止在无法建立原生事务时退化执行多条独立 SQL。连接字典重建后仍失败会在任何业务写入前终止操作并提示重新打开 App，避免方案、购买账单或备份导入只保存一半。
- 修复 Markdown 报告读取账单初始 `pending/returnCents=0` 的问题：导出时现在使用最新赛果重新计算每笔冻结方案，动态输出进行中/已完成和当前回款。
