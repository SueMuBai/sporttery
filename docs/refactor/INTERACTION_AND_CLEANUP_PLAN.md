# 彩果 App 第二阶段：旧代码清理与移动端交互重构计划

> 编制日期：2026-07-18
> 适用基线：Vue 3 + TypeScript + Vite + Pinia + Vant + Capacitor 7 重构版
> 目标：清除旧架构遗留，重新梳理业务状态与页面交互，使应用从“网页功能集合”转为一致、可预测的移动端 App。

## 1. 当前结论

当前技术框架无需再次推翻。Vue、Pinia、Vue Router、Vant、Capacitor 和 SQLite 足以支撑长期维护；现阶段的主要问题不是框架能力，而是第一阶段为了快速覆盖旧功能，直接把多个网页式表单、弹窗和业务概念平移到了新页面。

最明显的问题是：

- “已购”被实现为普通标签，但它实际上是会创建账单的业务动作。
- 保存方案时同时要求处理名称和标签，主任务被附加配置打断。
- 方案管理把查看、改名、标签、导入、合并、删除都堆在多层 Bottom Sheet 中。
- 页面大量依赖弹层完成编辑，缺少移动端常见的“列表进入详情、详情进入编辑”层级。
- 筛选、排序、刷新、同步等入口在不同页面采用不同组件和反馈方式。
- 删除、切换玩法、覆盖选票等确认策略没有统一风险等级。
- 一些 Vant 默认图标、字段、下拉菜单仍保留明显 Web 表单感。

本阶段不以“继续补样式”为主，而是先修正领域模型和任务流，再统一组件与视觉。

## 2. 重构原则

### 2.1 一个动作只表达一个业务意图

- 保存方案：只保存当前选择，默认名称自动生成，用户可选改名。
- 记录购买：明确创建账单，需要确认实际投入、倍数、购买时间。
- 标签：只用于分类和筛选，不触发购买、结算或其他业务副作用。
- 导入方案：只负责把方案载入选票，不顺带修改原方案。
- 编辑方案：明确覆盖原方案或“另存为”，不能默默改变保存记录。

### 2.2 高频动作一步完成，低频配置进入二级页

- 高频：选择赔率、清空、保存、查看当前选择、刷新。
- 中频：记录购买、导入方案、修改名称、调整标签。
- 低频：标签新增/改名/排序、同步参数、导出与诊断工具。

### 2.3 同一页面最多保留一层临时浮层

- Bottom Sheet 用于短选择和轻确认。
- 需要输入多个字段、展示大量信息或继续跳转的任务使用二级页面。
- 禁止 Sheet 内再打开 Sheet、日历或复杂编辑器。
- 系统确认框只用于不可逆或高风险操作。

### 2.4 业务状态与展示标签彻底分离

建议的数据概念：

| 概念 | 作用 | 示例 |
| --- | --- | --- |
| 方案状态 | 系统计算，不由用户随意填写 | 草稿、已保存、已购买、已完成 |
| 账单状态 | 由比赛与回款决定 | 进行中、已结算 |
| 用户标签 | 仅分类 | AI、稳健、冷门、世界杯 |
| 同步状态 | 反映数据新鲜度 | 未同步、同步中、已更新、部分失败 |

“已购”不得作为决定是否创建账单的标签，也不预置该标签。购买状态必须依据账单关联关系判断。

## 4. 新的核心业务模型

### 4.1 方案生命周期

```text
选票临时选择
  ├─ 保存方案 → 已保存方案
  ├─ 直接购买 → 已保存方案 + 账单记录
  └─ 清空/离开 → 保留本地草稿，用户确认后才丢弃

已保存方案
  ├─ 载入选票 → 临时编辑副本
  ├─ 覆盖保存 → 更新原方案
  ├─ 另存为 → 创建新方案
  ├─ 记录购买 → 新增账单快照
  └─ 删除 → 删除方案，既有账单快照不变
```

### 4.2 保存与购买分离

#### 保存方案

默认交互：点击“保存方案”后立即以自动名称保存并提示成功。

- 自动名称：`周四 8场 2关 · 07-18 23:50`。
- 成功提示提供短时操作：“查看”或“改名”。
- 不要求填写标签。
- 若当前选票来自已有方案，按钮文案改为“保存修改”，并提供“另存为”。
- 内容未变化时禁用重复覆盖保存。

#### 记录购买

独立入口：当前方案区域的“记录购买”。

需要确认：

- 方案名称，只读或可轻量修改。
- 实际投入金额，默认按注数与倍数计算。
- 倍数。
- 购买时间，默认当前时间。
- 可选备注。

确认后创建冻结的账单快照。标签不参与创建账单。

### 4.3 标签规则

- 标签只能在“设置 → 标签管理”新增、改名、调整颜色、排序和删除。
- 保存方案页、方案详情页只能从已有标签中选择，不提供文本创建入口。
- 新保存方案默认不要求选择标签。
- 方案详情中的“管理标签”打开单层选择 Sheet；无标签时只显示“前往标签管理”。
- 删除标签仅解除当前方案关联，不改变账单快照。
- 系统状态不得伪装成标签，也不得预置“已购”等状态型标签。

## 5. 分页面交互重构

## 5.1 选票页

### 页面结构

1. 顶部标题与单个刷新按钮。
2. 玩法切换栏。
3. 搜索与轻量筛选。
4. 比赛列表。
5. 底部常驻选票摘要。

### 交互规则

- 刷新按钮一次完成“获取最新比赛 + 更新已有赛果”，结果在同步报告中拆分显示。
- 玩法切换只改变浏览内容，不清除已选项。
- 同一比赛选择另一玩法时，使用就地提示条说明会替换已有玩法；只有已有多项选择会被清除时才确认。
- 比赛卡展开历史交锋时在卡片内部展开，不跳弹窗。
- 底部摘要默认显示场次、注数、金额；上拉后显示过关方式、倍数、理论奖金和操作。
- 主要操作固定为“保存方案”和“记录购买”。
- “查看方案”改名为“查看当前选择”，只展示当前临时选票，不与已保存方案混淆。
- 清空属于中风险操作：无选择时禁用；有选择时一次确认，确认文案明确清除场次和选项数量。
- 离开选票时自动保留草稿，不因切换底部导航丢失。

### 状态反馈

- 首次无数据：引导刷新。
- 有旧数据且刷新失败：继续展示旧数据，用非阻断提示说明离线。
- 同步中：刷新按钮旋转，页面仍可浏览，避免全屏 Loading。
- 局部历史请求失败：只在对应卡片重试。

## 5.2 保存方案与当前选择

取消现有“名称字段 + 标签列表 + 确认保存”的强制长表单。

新流程：

1. 点击保存。
2. 自动命名并保存。
3. Toast/轻提示显示“方案已保存”，附“查看”操作。
4. 用户在详情页按需改名或设置已有标签。

如产品最终仍要求保存前命名，使用单字段小 Sheet，仅包含名称和保存按钮；标签仍不得混入该步骤。

当前选择详情使用独立二级页或全屏 Sheet，内容包括：

- 已选比赛与选项。
- 过关方式、倍数、注数、金额、奖金区间。
- 删除单场、回到选票定位比赛。
- 保存方案、记录购买。

## 5.3 方案管理

### 列表页

- 顶部搜索。
- 一个“筛选”入口，打开单层筛选 Sheet；不再同时堆状态胶囊和三个 Web 下拉菜单。
- 筛选条件：购买状态、结算状态、用户标签、过关方式、排序。
- 已生效筛选以紧凑 Chip 显示，可单独移除。
- 点击卡片主体进入详情；卡片上的直接动作最多保留一个“载入选票”。
- 更多操作统一放入卡片右上角菜单，且菜单不再继续套编辑 Sheet。

### 详情页

详情页是方案的唯一管理中心：

- 顶部：名称、系统状态、用户标签。
- 摘要：投入估算、场次、注数、过关、倍数、奖金区间。
- 比赛选项与赛果。
- 底部操作：载入编辑、记录购买。
- 更多：改名、修改标签、复制方案、删除。

改名可以在详情页标题区域内联完成。标签使用单层选择 Sheet。删除使用系统确认框。

### 导入/编辑规则

- “载入编辑”默认创建临时编辑副本，不立即改数据库。
- 当前选票为空：直接载入。
- 当前选票有未保存变化：提供“保存当前后载入 / 放弃当前并载入 / 取消”。
- 取消“合并为新方案”这一含义不清的快捷动作。
- 若需要合并，作为独立高级功能设计：先展示冲突清单，再逐场处理，不允许一键静默合并。

## 5.4 账单页

- 顶部日期按钮打开唯一日期筛选 Sheet，日历和快捷范围在同一层。
- 页面默认当前月，筛选状态始终在统计周期卡中可见。
- 账单卡点击进入独立详情路由，避免用大型 Bottom Sheet 承载完整方案。
- 进行中账单显示“预计回款/当前已结算回款”，不得叫最终回款。
- 全部完成后，回款金额在详情页内联编辑；列表页只展示结果。
- 手工回款必须显示“手工调整”标识，并保留自动计算值作为参考。
- 账单详情以购买时快照为准；后续方案改名或删除不能改变快照内容。
- 增加账单备注编辑、手工回款历史和撤销最近一次修改的能力。

日期、金额、收益术语统一：

| 场景 | 文案 |
| --- | --- |
| 尚有比赛未完成 | 当前已结算回款 |
| 全部完成且未手工修改 | 理论回款 |
| 用户填写后 | 实际回款 |
| 实际回款 - 投入 | 实际盈亏 |

## 5.5 设置页

- 一级页只保留导航，不直接执行耗时导出或同步。
- 系统设置、标签管理、数据更新、方案管理、数据与备份、关于分别进入二级页。
- Markdown/JSON 合并为“数据与备份”，在二级页选择导出格式、范围和保存位置。
- 设置项按“自动保存”和“点击保存”二选一统一；推荐数值配置修改后点击页面底部保存。
- 未保存离开时统一提示，不由各页面自行实现不同文案。
- 标签管理采用列表 + 二级编辑页或单字段 Sheet；颜色显示友好名称和色块，不直接把十六进制颜色作为按钮文本。
- 标签排序优先拖拽；若拖拽真机体验不稳定，再保留上下移动作为无障碍替代。

## 5.6 数据更新页

- 首页刷新与设置内“数据更新”调用同一个 SyncController。
- 展示最后成功同步时间、比赛新增数、赔率变化数、赛果新增/更新数、失败数。
- 正在同步时支持离开页面，完成后通过应用内通知提示。
- 部分失败可针对失败项重试，不重新请求全部数据。
- 不把技术字段、接口 URL 和原始异常直接展示给普通用户；详细日志放到“诊断信息”。

## 6. 统一交互组件

需要新增或重构以下公共组件：

| 组件 | 职责 |
| --- | --- |
| `AppPage` | 安全区、标题区、滚动区、底部导航统一布局 |
| `AppTopBar` | 一级/二级页标题、返回、单一主操作 |
| `AppBottomSheet` | 标题、关闭、内容、固定操作区、Android 返回行为 |
| `AppListGroup` | 设置与管理页分组列表、分割线、箭头 |
| `AppFormRow` | 标签、值、说明、错误与点击区域 |
| `AppFilterSheet` | 筛选草稿、应用、重置，不直接边点边关闭 |
| `AppConfirmAction` | 危险等级、统一按钮顺序和文案 |
| `AppInlineEditor` | 改名、回款、备注的就地编辑 |
| `AppSyncIndicator` | 同步中、成功、部分失败和最后更新时间 |
| `AppEmptyState` | 空数据、无筛选结果、离线数据三类状态 |

组件必须满足：

- 点击区域至少 44×44dp。
- 按钮文字与图标真正居中。
- Android 返回键优先关闭当前唯一浮层，再返回路由。
- 焦点、禁用、加载、错误、选中状态一致。
- 业务页不得直接覆盖 Vant 内部 DOM 来实现关键布局。

## 7. 路由与浮层规划

建议路由：

```text

/ledger/:orderId
/ticket
/ticket/current
/plans
/plans/:planId
/settings
/settings/system
/settings/tags
/settings/tags/:tagId
/settings/sync
/settings/data
/settings/about
```

保留为浮层的内容：

- 日期筛选。
- 简单筛选。
- 已有标签多选。
- 单字段改名。
- 危险操作确认。
- 简短操作菜单。

必须使用页面的内容：

- 当前选择完整详情。
- 方案详情。
- 账单详情。
- 系统配置。
- 标签管理列表。
- 数据导出与备份。
- 同步报告和诊断信息。

## 8. 数据库调整

建议 schema v5：

1. 方案增加 `sourcePlanId`、`revision`、`status`，支持编辑副本和覆盖检测。
2. 账单增加 `actualReturnCents`、`returnSource`、`settledAt`、`notes`。
3. 新增 `ledger_adjustments`，保存回款修改历史，而不是只覆盖最终值。
4. 方案与标签继续使用关联数据，不将标签文本作为业务开关。
5. 新增 `draft_ticket` 或通用 draft 表，保存当前选票草稿和来源方案。
6. 新增 `app_notifications`，用于后台同步完成提示。
7. 依据 `ledger_orders.planId` 计算方案是否购买，不读取任何标签文本判断状态。

当前重构直接使用全新 schema，不承担旧版本升级迁移。数据库行为仍需覆盖：一个方案零次或多次购买、方案删除后账单快照保留、标签删除不影响方案与账单等当前模型约束。

## 9. 执行阶段

### 阶段 A：冻结行为与建立交互规范

- 为当前保存、购买、标签、导入、回款流程补充行为测试。
- 输出领域状态图、路由图、浮层规则和统一文案表。
- 建立真机验收截图基线。

验收：旧行为被测试记录，后续改动能明确判断是有意变化还是回归。

### 阶段 B：旧代码安全清理

- 删除 Android 模板测试和确认未引用资源。
- 清理未使用的 Vue 代码、样式和 Vant 覆盖。
- 删除旧数据文件和迁移兼容层。
- 更新 README、进度和当前数据模型文档。

验收：类型、Lint、单测、生产构建、Android 同步全部通过；旧用户数据未被修改。

### 阶段 C：领域模型和全新数据库

- 拆分方案状态、购买记录和用户标签。
- 增加草稿、版本、回款调整历史。
- 为全新 schema 编写数据库行为和事务恢复测试。

验收：方案、标签、账单边界清晰，金额与冻结快照稳定，“已购”不再触发业务副作用。

### 阶段 D：统一页面骨架和交互组件

## 10. 必测场景

当前逐项状态与可复现证据见 [ACCEPTANCE_MATRIX.md](./ACCEPTANCE_MATRIX.md)。

1. 无标签时保存方案成功。
2. 新增标签只能从标签管理完成。
3. 保存方案不会创建账单。
4. 记录购买一定创建冻结账单快照。
5. 同一方案可以多次购买并形成多笔独立账单。
6. 删除方案不删除历史账单。
7. 删除标签不删除方案或账单。
8. 载入方案前保护当前未保存选票。
9. 覆盖保存与另存为结果明确且可测试。
10. 离线刷新失败后仍能使用本地比赛。
11. 进行中账单只显示当前已结算回款。
12. 手工回款后保留原计算值和修改历史。
13. Android 返回键不会一次退出多个页面或留下遮罩。
14. 键盘弹起时底部主按钮可见且可操作。
15. 115% 字体下按钮文本、金额和标签不溢出。

## 12. 完成定义

本阶段完成不以“页面看起来更像 App”为唯一标准，必须同时满足：

- 方案、购买、账单、标签四个概念边界清晰。
- 用户完成高频任务不需要填写无关信息。
- 页面之间没有重复入口和相互矛盾的动作。
- 不存在以标签文本驱动核心业务的逻辑。
- 大型内容使用路由页面，临时选择使用单层 Sheet。
- 旧运行时、旧数据文件和迁移兼容层已彻底移除。
- 全部关键流程有自动测试和 Android 真机验收记录。
